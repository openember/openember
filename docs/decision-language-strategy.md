# OpenEmber 语言路线决策报告（C + C++ vs C++ 主线）

## 1. 结论先行（建议）

建议采用：

- **产品与生态层面：以 C++ 为主线（默认、优先、文档主路径）**
- **兼容与集成层面：保留稳定 C ABI 边界，而不是维护“完整双实现双模板”**

换句话说，不建议当前阶段继续投入“C 与 C++ 两套完整同构模板/框架并行演进”；建议收敛为：

- 核心 runtime、调度、插件、配置、观测工具链优先 C++
- 对外暴露一层精简且稳定的 C 接口，服务历史项目、RTOS/裸驱动包装、以及对 C 语言有硬约束的场景

这是在“可持续演进速度”和“工业可落地性”之间最稳妥的中间路线。

---

## 2. 为什么现在要做这个决策

当前仓库已经出现典型信号：

- 目标愿景很大（分布式、插件化、多中间件、多模块）
- 工程面尚在快速迭代（目录、CMake、组件边界都在演进）
- 若再并行维护 C/C++ 两套主线，会明显放大复杂度、降低交付速度

你担心的点非常准确：**双主线不是“多写一点代码”，而是“指数级增加决策面和维护面”。**

---

## 3. 方案对比

### 方案 A：继续 C 与 C++ 双主线并行（完整双模板）

优点：

- 理论覆盖面最广（MISRA-C、传统团队、现代 C++ 团队都可直用）
- 宣传层面“兼容性强”

缺点（核心）：

- 需求实现要做两次（功能、测试、文档、示例、CI、发布）
- API 一致性成本极高，长期会“名义一致、行为分叉”
- 中间件抽象、错误模型、生命周期管理在两语言里很难长期等价
- 维护者精力被稀释，迭代速度显著下降

适用前提（目前并不满足）：

- 稳定的核心架构 + 充足维护人力 + 严格自动化测试体系

### 方案 B：C++ 主线 + C ABI（推荐）

优点：

- 统一内部实现，复杂能力（异步、插件、可观测）优先落地
- 对存量 C 生态保持可接入能力
- 文档、示例、CI 主路径单一，节奏更快
- 架构演进时，C 侧只维护“稳定边界”，维护成本可控

风险：

- 纯 C 深度定制能力会受限（需要通过 C ABI 间接使用）
- 若 ABI 设计不严谨，会出现可用性问题

应对：

- C ABI 只承诺“高价值、低变化”的最小能力集
- 明确哪些能力仅 C++ API 支持（插件系统、高级异步链路等）

### 方案 C：纯 C++（完全不保留 C 入口）

优点：

- 理论复杂度最低

缺点：

- 会直接丢掉一部分嵌入式传统生态与集成场景
- 对“工业现场已有 C 资产”迁移不友好

结论：

- 对 OpenEmber 这种“面向设备和工业项目”的定位，纯 C++ 过于激进。
- **B 方案是现实最优解。**

---

## 4. 结合 ROS/AimRT 的参考判断

- ROS2 社区主流开发体验确实更偏 C++（核心能力和生态工具链围绕 C++ 很强），但也保留 C 相关接口层，说明“主线 + 边界”是工程上可行的长期策略。
- AimRT 明确定位现代 C++ runtime，强调异步、插件和部署能力，这与“先把核心能力在一个强表达力语言里做深”一致。

参考：

- [AimRT GitHub](https://github.com/AimRT/AimRT)
- [AimRT 文档首页](https://en.docs.aimrt.org/index.html)

注意：OpenEmber不应“照抄”AimRT，而应借鉴其路线选择逻辑：**先形成强主线，再通过边界兼容其他生态**。

## 5. 建议的技术边界（关键）

### 5.1 分层建议

- `runtime-core`（C++）：节点生命周期、调度、插件装载、配置分发、观测 hooks
- `transport-*`（可 C/C++）：MQTT/ZMQ/NNG 等适配器
- `openember-c-api`（C）：稳定 ABI 门面（handle + 函数表）
- `openember-cpp-api`（C++）：完整能力

### 5.2 C ABI 设计原则

- 不暴露 C++ 对象，统一 `opaque handle`
- 错误码稳定、线程语义清晰
- 生命周期函数成对：`create/init/start/stop/destroy`
- 消息结构优先用 plain struct + size，避免 ABI 脆弱类型
- 所有回调都可携带 `user_data`

### 5.3 能力分级（建议公开）

- **L1（C/C++ 都支持）**：pub/sub 基础、配置读写、日志、健康状态
- **L2（仅 C++）**：插件热加载、复杂异步编排、高级观测、策略化调度

## 6. 对当前仓库的落地路线（3 阶段）

### 阶段 1：路线收敛（1~2 周）

- 冻结“新增双模板”动作
- 明确默认主线：C++（文档、示例、CI）
- 定义最小 C ABI 草案（先覆盖 pub/sub + node lifecycle）

交付物：

- `docs/language-policy.md`（规范）
- `include/openember/c_api.h`（草案）
- `examples/cpp/*` 与 `examples/c/*`（仅保留一对最小示例）

### 阶段 2：能力归核（2~4 周）

- 将现在的通信骨架统一到 runtime 抽象下
- C API 通过门面调用 C++ runtime
- 建立 ABI 兼容检查（至少符号级别）

交付物：

- `runtime-core` 初版
- `transport` 统一接口
- CI 增加 C API 示例编译与运行测试

### 阶段 3：生态定型（持续）

- 脚手架 `ember new` 默认生成 C++ 工程
- `ember new --lang c` 生成“C API 接入工程”而不是独立 C runtime
- 文档将“双路线”改为“主线 + 兼容层”叙事

---

## 7. 决策守则（避免后续反复摇摆）

以后任何新增需求，先问三件事：

1. 这个能力是否必须在 C 侧“原生实现”？  
2. 如果不是，是否可通过 C ABI 暴露“可用子集”？  
3. 若必须双实现，是否有对应维护者和测试预算？

若第 3 条不满足，默认不做双实现。

---

## 8. 最终建议（可直接用于项目会议）

- **战略决策**：采用“C++ 主线 + 稳定 C ABI”。
- **执行策略**：暂停双模板扩张，先把主线做深做稳。
- **对外表述**：OpenEmber 支持 C 与 C++ 接入，但核心 runtime 以 C++ 实现，C 通过稳定 ABI 使用核心能力。

这能最大化你们当前阶段最稀缺的资源：**核心团队注意力与迭代速度**，同时保留工业项目最看重的兼容性与可落地性。
